# 智慧云脑诊疗平台 - 医生端微服务拆分与 DDD 战略设计文档

**文档概述：**
本文档基于“智慧云脑诊疗平台”业务模块图，采用领域驱动设计（DDD）方法论，对**医生端**的核心业务流程进行自顶向下的战略设计与微服务边界划分。核心目标是实现系统的高内聚、低耦合，确保各微服务拥有独立的数据模型与演进生命周期。

---

## 1. 架构风格与拆分原则

* **设计方法：** 领域驱动设计 (Domain-Driven Design)
* **拆分原则：** 以业务能力和限界上下文 (Bounded Context) 为边界，单一微服务不跨越上下文。
* **集成方式：** 弱同步依赖，强异步解耦。上下文间主要通过**领域事件 (Domain Events)** 进行交互。**关键业务操作（医嘱下发、处方提交）采用同步确认 + 异步执行的两阶段模式**：下游服务同步返回 `accepted` 状态后，上游才向用户反馈成功；后续执行流转通过领域事件异步完成。
* **基础设施横切关注点：** 分布式日志收集、链路追踪、指标监控作为全系统共享的基础设施层，各微服务统一接入，不单独建模为限界上下文。

## 2. 限界上下文 (Bounded Context) 划分

### 2.1 门诊工作台上下文 (Outpatient Consultation Context)
* **领域类型：** 核心域 (Core Domain)
* **业务边界：** 统筹医生接诊患者的完整生命周期（从呼号接诊到结束就诊），整合病历阅览、问诊记录、下达医嘱和开具处方。该上下文是医生端的“枢纽”，仅负责发出业务指令，不关心底层执行细节。
* **核心聚合根 (Aggregate Root)：**
    * `就诊记录 (Consultation)`
    * `电子病历 (EMR)`
* **核心动作：** 开启就诊、更新病历、开具医嘱、结束就诊。

### 2.2 医学影像与 AI 处理上下文 (Medical Imaging & AI Analysis Context)
* **领域类型：** 支撑域 (Supporting Domain)
* **业务边界：** 接收医技科室上传的原始影像数据，执行标准化预处理与 AI 算法推理，生成结构化报告。
* **核心聚合根：**
    * `影像分析任务 (Imaging Task)`
    * `智能检查报告 (AI Report)`
* **领域职责：** 处理高密度的计算任务与复杂的医学数据格式。系统需解析传入的 DICOM (`.dcm`) 序列或 NIfTI (`.nii.gz`) 体数据，完成张量化预处理后，调度底层推理引擎（如通过 Vertex AI 托管的模型）获取结果，并组装为报告实体。

### 2.3 AI 助理医生上下文 (AI Assistant Doctor Context)
* **领域类型：** 支撑域 (Supporting Domain)
* **业务边界：** 作为一个独立的 Agent 运行平台，提供基于大语言模型的智能交互与文本结构化服务。
* **核心聚合根：**
    * `辅助诊断会话 (Diagnostic Session)`
* **交互模式：** 问诊文本通过 **WebSocket 长连接**从前端直达本服务，不经过 Kafka 中转。非实时的历史病历文本通过领域事件异步投递。
* **领域职责：** 对问诊对话流进行实时意图识别、实体抽取与术语标准化，根据预设的提示词链与 Agent 脚本向医生工作台推送鉴别诊断和用药建议。就诊结束后，发布 `ConsultationAiSummaryEvent` 将结构化摘要异步写入病历。

### 2.4 排班与队列上下文 (Scheduling & Queue Context)
* **领域类型：** 支撑域 (Supporting Domain)
* **业务边界：** 管理医生的时间资源（出诊计划）和患者的就诊顺序（融合线上与线下挂号）。
* **核心聚合根：**
    * `排班计划 (Schedule)`
    * `诊室队列 (Queue)`
* **领域职责：** 维护队列状态机的流转（候诊中、就诊中、已过号），为门诊工作台提供“下一个患者”的调度指令。

### 2.5 医嘱执行与路由上下文 (Order Execution & Routing Context)
* **领域类型：** 通用域 (Generic Domain) / 支撑域
* **业务边界：** 充当门诊工作台与外部医技科室（检查/检验、药房）之间的防腐层 (Anti-Corruption Layer)。提供同步 RPC 端点供门诊工作台提交医嘱，内部持久化后立即返回 `accepted`，随后异步发布事件驱动下游执行。
* **核心聚合根：**
    * `医嘱单 (Medical Order)`：维护医嘱状态机（`pending` → `accepted` → `routed` → `completed` / `rejected`）
* **领域职责：** 接收门诊开具的逻辑医嘱，持久化并同步确认接收，然后将其转换为具体的计费项目和执行单据，通过领域事件异步路由分发到对应的下游系统。

### 2.6 认证授权上下文 (Authentication & Authorization Context)
* **领域类型：** 通用域 (Generic Domain)
* **业务边界：** 作为系统安全基座，统一管理用户身份认证、角色权限与令牌生命周期。所有前端请求经 API Gateway 转发至本上下文进行 JWT 校验，各微服务在领域逻辑中根据令牌内携带的角色与权限进行数据级访问控制。
* **核心聚合根：**
    * `用户 (User)`：关联唯一登录凭证、角色列表与所属科室（适用于医生/医技人员）
    * `角色 (Role)`：定义 RBAC 权限集合（Patient / Doctor / Technician / Admin）
* **领域职责：**
    * 颁发与刷新 JWT 访问令牌（Access Token + Refresh Token）
    * 维护用户-角色-权限的绑定关系
    * 提供令牌内省端点供 Gateway 做集中鉴权
    * 发布 `UserLoggedInEvent`、`PermissionChangedEvent` 等安全审计事件

## 3. 上下文映射与事件集成 (Context Mapping & Integration)

微服务之间通过事件总线进行异步通信，避免微服务间的 RPC 级联故障。

### 3.1 认证授权与全局鉴权流转
* **同步交互（入口鉴权）：** API Gateway 将所有请求转发至认证授权上下文进行令牌校验。校验通过后，Gateway 将解析出的 `principal`（用户 ID + 角色列表）注入请求头，下游微服务直接信任该身份上下文。
* **异步事件（安全审计）：**
  * `UserLoggedInEvent` — 用户登录事件，供门诊工作台上下文初始化医生偏好配置
  * `PermissionChangedEvent` — 权限变更事件（如医生被停权），各消费端需在本地缓存中即时失效对应令牌

### 3.2 核心事件流转示例 (以 AI 影像报告为例)
1.  **影像上下文**完成模型推理与报告组装。
2.  **影像上下文**发布集成事件：`CtReportGeneratedEvent` (CT 报告已生成事件)。
3.  **门诊工作台上下文**订阅该事件，接收到通知后，在前端工作台推送状态变更（如弹窗或红点），提示医生可查阅影像和报告。

### 3.3 共享内核 (Shared Kernel) 与防腐策略
为保证跨服务事件交互的契约严谨性，特别是在强类型语言生态下开发微服务时，采用 Shared Kernel 模式：
* **独立包管理：** 将集成事件的数据结构（如 `CtReportGeneratedEvent`、`UserLoggedInEvent`）、状态枚举、以及跨上下文的 DTO 对象提取为独立的 Maven 契约包。
* **依赖引入：** 利用 Maven 将契约包统一分发。生产者与消费者服务分别引入该包，从而在编译期严格保证反序列化时的类型安全，防止微服务间的领域模型被相互侵入。

### 3.4 医嘱下发的同步确认与异步执行（两阶段模式）

医嘱是医疗安全的关键链路，不能采用"发事件即返回"的纯异步模式。系统采用**同步确认 + 异步执行**的两阶段设计：

```
[门诊工作台上下文] ──POST /orders (同步 RPC)──➔ [医嘱执行上下文]
       │                                              │
       │                                    ┌──── 持久化医嘱单 ────┐
       │                                    │  (状态: accepted)    │
       │                                    └─────────────────────┘
       │                                              │
       ◄──────── 返回 {order_id, status: "accepted"} ─┘
       │
       │  (医生端看到"医嘱已提交"确认)
       │
       ▼ (异步阶段 — 之后的事情)
[医嘱执行上下文] ──OrderIssuedEvent──➔ [Kafka] ──➔ [药房 / 检查科室]
                                                     │
                                              (执行完成后回传状态)
```

1. **同步阶段（必需）：** 门诊工作台通过 OpenFeign 同步调用医嘱路由服务的 REST 端点。路由服务在本地事务中持久化医嘱单（状态 `accepted`），立即返回 `order_id`。若调用超时或返回错误，门诊前端即时提示医生重试，**杜绝医嘱静默丢失**。
2. **异步阶段（后续）：** 路由服务通过 Outbox + Kafka 发布 `OrderIssuedEvent`，下游药房、检查科室消费该事件执行实际配药或检查预约。执行完成后通过 `OrderFulfilledEvent` 回写医嘱状态。
3. **超时与重试：** 门诊端 Feign 客户端配置 3s 连接超时 + 5s 读超时。超时后医生端展示"医嘱提交中，请稍候"，并支持手动重试（幂等：同一 `consultation_id` + 业务内容 hash 去重）。

### 3.5 AI 助理的实时通信与异步归档

AI 助理医生的交互分为实时和离线两条通道，分别采用不同的通信方式：

| 通道 | 内容 | 通信方式 | 说明 |
|---|---|---|---|
| **实时上行** | 医生问诊文本 | WebSocket 前端 → AI 助理 | 低延迟流式传输，实时推送诊断建议 |
| **实时下行** | 鉴别诊断、用药提醒 | WebSocket AI 助理 → 前端 | 秒级推送，直接展示在医生工作台侧栏 |
| **离线归档** | 会话结构化摘要 | Kafka `ConsultationAiSummaryEvent` | 就诊结束后异步写入病历，不阻塞实时交互 |

**设计依据：** 实时问诊需要低延迟的点对点传输，Kafka 的持久化-消费模型会引入不可接受的延迟和资源开销。而就诊结束后的摘要归档是离散的异步任务，天然适合 Kafka 事件驱动。
