# 智慧云脑诊疗平台 - 医生端微服务架构与 DDD 落地设计方案

本项目针对“智慧云脑诊疗平台”的医生端业务进行微服务划分，整体设计遵循**领域驱动设计（DDD）**方法论。为了适应小规模项目的敏捷开发与低运维成本要求，架构方案**完全废弃了复杂的 Kubernetes (K8s) 编排**，存储层**统一收敛至 MySQL**，并引入**轻量化 Kafka（KRaft 模式）**作为事件总线，构建出一套高内聚、低耦合、技术栈高度关联的事件驱动微服务架构（EDA）。

---

## 1. 架构设计原则与选型核心

1. **逻辑与物理边界对齐：** 严格按照 DDD 限界上下文（Bounded Context）划分微服务。每一个微服务都是一个独立的业务领域，拥有自治的数据模型和生命周期。
2. **轻量化运维（去 K8s 化）：** 拒绝引入复杂的容器编排平台，采用进程级/标准容器级部署，通过轻量级网关与服务发现组件进行内部寻址。
3. **MySQL 主存储 + Elasticsearch 检索引擎：** MySQL 作为唯一的事务写入主库，承担所有 ACID 操作（就诊记录、医嘱、排班、用户）。**电子病历文本、AI 报告、检查结果**等需要全文检索的数据，写入 MySQL 后通过 Outbox + Kafka 异步同步至 Elasticsearch。ES 仅作为只读检索引擎，不承担写入职责，保持单一写入入口。
4. **事务性事件驱动：** 微服务间日常通过 Kafka 进行异步解耦。使用**事务发件箱模式（Transactional Outbox Pattern）**消除分布式事务难题。**关键业务链路（医嘱下发）采用同步 RPC 确认 + 异步事件执行的两阶段模式**，在解耦与临床安全之间取得平衡。

## 2. 医生端限界上下文划分与微服务拆分

### 2.1 门诊工作台微服务 (Outpatient Consultation Service)
* **领域类型：** 核心域 (Core Domain)
* **业务边界：** 负责医生接诊患者的完整生命周期管理。包含患者叫号接收、历史病历阅览、问诊过程记录、下达医嘱、开具处方以及结束就诊。它是整个医生端的核心业务枢纽。
* **核心聚合根：** * `就诊记录 (Consultation)`：维护问诊状态机（候诊、就诊中、已过号、已结束）。
  * `电子病历 (EMR)`：归属于特定的就诊记录，包含结构化的主诉、现病史、体格检查等。
* **数据存储策略：**
  * **写路径：** 就诊状态、电子病历（含动态模板的 JSON 内容）统一写入 MySQL，在同一事务中插入 Outbox 事件，保证 ACID。
  * **读路径（搜索）：** 电子病历的全文检索（主诉、现病史、体格检查等长文本）走 Elasticsearch——支持 IK 中文分词、BM25 相关性排序和同义词扩展。MySQL 仅按主键/外键做精确查询（如"按就诊 ID 取病历"）。
  * **同步：** MySQL → Outbox → Kafka → 门诊工作台消费 → 索引至 ES，延迟通常在 500ms 以内。

### 2.2 医学影像与 AI 处理微服务 (Medical Imaging & AI Analysis Service)
* **领域类型：** 支撑域 (Supporting Domain)
* **业务边界：** 承接放射科等医技科室上传的头部 CT 等原始影像数据，调度后台 AI 推理引擎，执行影像伪影修复与病灶识别，最终生成结构化的智能检查报告。
* **核心聚合根：**
  * `影像分析任务 (Imaging Task)`：跟踪从接收序列、脱敏、算法推理到报告生成的全链路状态。
  * `智能检查报告 (AI Report)`：包含 AI 自动识别的病灶特征、定量参数及初步临床建议。
* **数据存储策略：** 原始的 DICOM (`.dcm`) 或 NIfTI (`.nii.gz`) 体数据文件存储于轻量级对象存储（如 MinIO 或本地挂载盘）中，**MySQL 仅记录文件绝对路径与 MD5 契约**。AI 结构化报告以 `JSON` 格式写入 MySQL，同时通过 Outbox 同步至 Elasticsearch，供医生按病灶特征、危机值等字段跨报告检索。

### 2.3 AI 助理医生微服务 (AI Assistant Doctor Service)
* **领域类型：** 支撑域 (Supporting Domain)
* **业务边界：** 作为一个独立的 AI Agent 运行工作台。在医生问诊时，前端通过 **WebSocket 长连接**将问诊文本发送至本服务，送入 LLM 进行医学意图识别、实体抽取，结果通过同一 WebSocket 连接推回医生工作台侧栏。
* **核心聚合根：** `辅助诊断会话 (Diagnostic Session)`（维护大模型上下文窗口、Prompt 链状态、WebSocket session 映射）。
* **离线归档：** 就诊结束后，AI 助理发布 `ConsultationAiSummaryEvent` 至 Kafka，由门诊工作台消费后写入病历归档。实时交互不走 Kafka，保证毫秒级响应。
* **特殊设计：** 该服务作为独立 Agent 平台运行，不直接操作核心业务表，通过 API Gateway 代理和特定的模型参数（如 LoRA 微调层、学习率微调策略）对接外部大模型（如 Google Vertex AI 或开源大模型）。

### 2.4 排班与队列微服务 (Scheduling & Queue Service)
* **领域类型：** 支撑域 (Supporting Domain)
* **业务边界：** 管理医生的出诊时间资源，并将“患者端”的线上挂号与现场的线下挂号数据进行融合，维护各个诊室的实时叫号队列。
* **核心聚合根：** `排班计划 (Schedule)`、`诊室队列 (Queue)`。

### 2.5 医嘱执行与路由微服务 (Order Execution & Routing Service)
* **领域类型：** 通用域 / 支撑域
* **业务边界：** 充当医生端与外部药房、计费、检查检验科室之间的**防腐层 (Anti-Corruption Layer)**。接收医生开具的逻辑医嘱，将其转化为具体的收费项目和科室执行单。

## 3. 技术栈高关联性选型

针对小规模项目的高效开发要求，选型极度强调组件之间的协同与精简：

| 层次 | 选型组件 | 关联性与关键作用描述 |
| :--- | :--- | :--- |
| **开发底座** | **Spring Boot 3.x + JDK 17/21** | 必须采用 JDK 17+。利用最新的**虚拟线程（Project Loom）**能力，每个 Kafka 消费者和门诊高并发请求都能分配到轻量线程，极大提升高并发叫号与问诊时的系统吞吐，无需配置复杂的线程池。 |
| **网关/风控** | **Spring Cloud Gateway** | 统一的请求入口。支持 HTTP 路由与 **WebSocket 代理**（AI 助理的实时文本经 Gateway 直通 AI 服务）。配置全局限流策略，保护后端 AI 服务防止大模型 API 被刷爆。 |
| **服务发现** | **Nacos (单机/轻量模式)** | 替代 K8s 内置的 Service 机制。集服务注册与配置中心于一体，各微服务通过 Nacos 实现平滑的内部 RPC 寻址（配合 OpenFeign）。 |
| **主存储层** | **MySQL 8.0+** | 唯一的写入主库。承载所有事务型数据（就诊记录、医嘱、排班、用户），利用 `JSON` 字段承载动态病历模板，通过 Outbox 表驱动数据变更事件。 |
| **检索引擎** | **Elasticsearch 8.x (单节点)** | 只读检索引擎。索引电子病历全文、AI 报告、检查结果，提供 IK 中文分词、BM25 相关性排序。写入仅通过 MySQL Outbox → Kafka 同步，不对业务代码暴露写接口。 |
| **事件总线** | **Apache Kafka 3.x+ (KRaft)** | 彻底**丢弃 Zookeeper**，采用原生的 KRaft 模式部署。Kafka 作为一个轻量级单容器运行，极大地减少了内存开销与运维复杂度，专门用于限界上下文之间的异步解耦。 |
| **实时通信** | **Spring WebSocket + STOMP** | AI 助理的实时文本与诊疗建议推送通道。前端通过 STOMP over WebSocket 与 AI 助理微服务直连，JWT 在 WebSocket 握手阶段校验，连接建立后双向流式传输。 |
| **认证授权** | **Spring Security + OAuth2 Resource Server + JWT** | 无状态认证方案。Gateway 层集中校验 JWT 签名与有效期，解析后的 `principal` 经请求头（`X-User-Id`、`X-User-Roles`）向下游微服务透明传播。高风险操作（开具处方、修改排班）在业务层叠加方法级权限注解（`@PreAuthorize`）进行二次拦截。 |
| **防腐层映射** | **MapStruct** | 在代码层面严格守住 DDD 上下文边界。用于在数据库实体（DO）、领域对象（Entity/VO）以及 Kafka 消息传输对象（DTO）之间进行安全、严谨、零反射的高性能对象转换。 |
| **链路追踪** | **Micrometer Tracing + OpenTelemetry** | 每个请求在 Gateway 入口生成全局 `traceId`，经 Feign 拦截器和 Kafka 消息头（`trace_id`）跨服务传播。所有追踪数据导出至 Zipkin，还原单次就诊的完整调用拓扑。 |
| **日志收集** | **Logback JSON + Fluentd + Elasticsearch + Kibana** | 各微服务以 JSON 格式输出结构化日志（含 `traceId`），Fluentd 作为 Daemon 容器收集并注入容器元信息后写入 Elasticsearch，Kibana 提供统一查询界面。 |

## 4. 认证授权架构设计

### 4.1 无状态 JWT 鉴权流

```
[客户端] ──POST /auth/login──➔ [API Gateway]
                                    │ (转发至认证服务)
                                    ▼
                            [认证服务：验证凭据]
                                    │
                          ┌─────────┴─────────┐
                          ▼                   ▼
                   [颁发 Access Token]   [颁发 Refresh Token]
                   (短期, 15 min)        (长期, 7 days, HttpOnly)
                          │
                          ▼
              [Gateway 全局过滤器校验 JWT]
                          │
              ┌───────────┼───────────┐
              ▼           ▼           ▼
          [门诊服务]   [影像服务]   [排班服务]
         (@PreAuthorize 方法级权限控制)
```

1. **集中鉴权：** 客户端携带 JWT 发起请求，API Gateway 的全局过滤器对令牌进行签名校验与有效期检查。无需各微服务重复实现鉴权逻辑。
2. **身份传播：** Gateway 校验通过后，将 `X-User-Id`、`X-User-Roles`、`X-User-Name` 以请求头形式注入下游调用，微服务信任该身份上下文，直接用于业务级权限判断。
3. **方法级授权：** 高风险操作（如 `@PreAuthorize("hasRole('Doctor')")` 标注处方开具接口）在微服务控制器层做二次拦截，防止令牌盗用后的横向越权。
4. **令牌刷新：** Access Token 过期后，客户端使用 Refresh Token 换取新的 Access Token。Refresh Token 存储于服务端（MySQL），支持主动吊销。

### 4.2 处方/医嘱的强授权

开具处方属于医疗安全关键操作，在 JWT 角色校验之外叠加二级确认机制：
* 医生点击"开具处方"时，前端弹出二次密码确认框。
* 认证服务提供 `/auth/verify-credential` 端点，Gateway 对该端点配置最高速率限制（每分钟 3 次），防止暴力破解。

### 4.3 认证授权微服务数据模型

认证微服务拥有独立的数据库（与业务库隔离），核心表：
* `user` — 用户登录凭据（bcrypt 加密）、关联角色、所属科室 ID
* `role` — 角色定义（Patient / Doctor / Technician / Admin）
* `refresh_token` — 已签发的 Refresh Token 哈希，支持按用户维度批量吊销

## 5. 核心难题攻克：数据与事件的最终一致性

在微服务架构中，向 MySQL 提交数据与向 Kafka 发送事件（双写）极易发生部分失败。由于去掉了 K8s 和重型分布式事务组件，系统采用**事务性发件箱模式 (Transactional Outbox Pattern)** 确保数据最终一致。

### 5.1 落地执行流（以 AI 影像报告生成为例）

```
[影像微服务本地事务]
 ├── 1. 更新影像任务状态为“已完成”
 └── 2. 插入一条事件到 `imaging_outbox` 表 (存储 CtReportGeneratedEvent JSON)
      └── ─── 提交事务 (MySQL 强一致性保证) ─── ┘
                                │
                        [定时异步轮询 / CDC]
                                │ (高频扫描 outbox 表未发送数据)
                                ▼
                    3. 将事件发布至 Kafka 集群
                                │
                        [Kafka 事件总线]
                                │ (Topic: medical-imaging-events)
                                ▼
                    4. 门诊工作台接收事件 (消费幂等)
                                │
                        [WebSocket 异步推送]
                                ▼
                    5. 医生端工作台界面亮起红点/弹窗
```

1. **原子写入：** 医学影像微服务处理完 CT 数据并生成智能报告后，在**同一个 MySQL 本地事务**中执行两个操作：
   * 将报告结果写入 `imaging_report` 表。
   * 将一条包含业务 ID 的 `CtReportGeneratedEvent` 记录插入到同库的 `imaging_outbox` 表中。
2. **可靠投递：** 微服务内部通过一个轻量级的 Spring 定时任务（推荐 200ms~1s 轮询间隔，平衡实时性与数据库负载）轮询 `imaging_outbox` 中未发送的事件，将其泵入 Kafka。投递成功后，将 outbox 表中的记录标记为已处理或删除。若对实时性有更高要求，可升级为 Debezium 等 CDC 方案，监听 MySQL binlog 实现毫秒级事件捕获。
3. **消费幂等：** 门诊工作台微服务订阅 Kafka 消息，收到通知后通过 WebSocket 异步推送到医生的前端界面。消费端通过事件中的全局唯一 `event_id` 或 `consultation_id` 进行幂等检查，防止消息重复消费，完美守住数据一致性的底线。

### 5.2 医嘱下发的同步确认实现

医嘱下发不适用纯异步 Outbox 模式——医生点击"开具"后必须即时知晓下游是否接收成功。该链路采用 **OpenFeign 同步 RPC + Outbox 异步执行** 的两阶段方案：

```
[门诊工作台]
     │ POST /api/orders (OpenFeign, 超时 3s connect + 5s read)
     ▼
[医嘱路由服务]
     │ @Transactional
     ├── 1. 幂等检查: 同一 consultation_id + content_hash 已存在则直接返回
     ├── 2. INSERT INTO medical_order (status='accepted', ...)
     ├── 3. INSERT INTO order_outbox (OrderIssuedEvent JSON)
     └── 提交事务 ───── 返回 {order_id, status: "accepted"}
     │
     ▼ (事务提交后)
[门诊工作台] 收到 200 ➔ 前端显示"医嘱已提交" ✓
     │
     ▼ (异步阶段 — 后续执行)
[Outbox 轮询] ➔ Kafka OrderIssuedEvent ➔ 药房/检查科室消费
```

| 环节 | 机制 | 关键参数 |
|---|---|---|
| **同步确认** | OpenFeign RPC 调用医嘱路由服务 | 连接超时 3s，读取超时 5s |
| **幂等保护** | 以 `consultation_id` + 业务内容 MD5 作为幂等键，重复提交返回原 `order_id` | — |
| **超时降级** | Feign 超时后前端展示"提交中"遮罩，医生可手动重试；后台异步查询订单状态 | 重试间隔 2s，最多 3 次 |
| **异步执行** | 医嘱路由服务 Outbox → Kafka → 下游消费 | Outbox 轮询 200ms~1s |

### 5.3 MySQL → Elasticsearch 异步索引同步

电子病历写入 MySQL 后，通过 Outbox + Kafka 自动同步至 ES，业务代码无感知：

```
[门诊工作台]
     │ @Transactional
     ├── INSERT INTO emr (JSON)         ← 病历写入 MySQL
     └── INSERT INTO emr_outbox         ← 同事务插入变更事件
              │
     [Outbox 轮询 200ms~1s]
              │
              ▼
     [Kafka: emr-data-change-events]
              │
              ▼
     [ES 索引消费者]                     ← 门诊工作台内部消费者
     ├── 构建/更新 ES 索引文档
     └── 标记 outbox 已处理
```

| 要点 | 设计 |
|---|---|
| **写入保证** | MySQL 事务中同时写入 EMR 和 outbox，ES 同步失败不影响主流程 |
| **ES 角色** | 纯只读检索引擎，业务代码永远不直接写 ES |
| **同步延迟** | 通常 < 500ms，医生保存病历后刷新即可搜索 |
| **索引策略** | ES 索引按 `科室` + `年月` 分片，每个科室独立的索引映射，支持动态字段扩展 |
| **重建索引** | 以 MySQL 为 source of truth，ES 索引损坏时全量重建即可 |

## 6. 关键业务事件流与 Kafka Topic 设计

通过提取一个独立的 **Shared Kernel (共享内核) Maven 模块**，将以下 Topic 的消息契约（DTO）集中管理，供所有微服务共同依赖，在编译期严格保证反序列化时的强类型安全。

### 6.1 `outpatient-consultation-events` (门诊业务主题)
* **生产者：** 门诊工作台微服务
* **消费者：** 医嘱执行微服务、AI 助理医生微服务
* **核心事件：**
  * `ConsultationStartedEvent`（问诊开始：AI 助理消费后初始化 WebSocket 会话上下文与 Prompt 链，准备接收问诊文本）。
  * `ConsultationFinishedEvent`（就诊结束：归档病历，触发离线医疗数据脱敏流程）。

### 6.2 `ai-assistant-events` (AI 助理主题)
* **生产者：** AI 助理医生微服务
* **消费者：** 门诊工作台微服务
* **核心事件：**
  * `ConsultationAiSummaryEvent`（AI 会话摘要：就诊结束后 AI 助理将结构化摘要——主诉提炼、鉴别诊断列表、用药建议——异步写入病历归档，不作为实时交互通道）。

### 6.3 `medical-order-events` (医嘱执行主题)
* **生产者：** 医嘱执行与路由微服务
* **消费者：** 药房管理、检查检验科室、门诊工作台微服务
* **核心事件：**
  * `OrderIssuedEvent`（医嘱已路由：医嘱路由服务同步确认接收后异步发布，包含已转换的计费项目和科室执行单，触发下游配药/检查预约）。
  * `OrderFulfilledEvent`（医嘱已执行：下游药房或检查科室完成执行后回传，门诊工作台消费后更新就诊状态）。
  * `OrderRejectedEvent`（医嘱被驳回：下游因库存不足、禁忌症等原因驳回时发布，门诊工作台消费后向医生弹窗告警）。

### 6.4 `medical-imaging-events` (医学影像与 AI 主题)
* **生产者：** 医学影像与 AI 处理微服务
* **消费者：** 门诊工作台微服务
* **核心事件：**
  * `CtReportGeneratedEvent`（头部 CT 智能报告已生成：包含初步诊断建议、危机值预警标识，门诊端消费后在医生工作台显著位置进行高亮提示）。

### 6.5 `doctor-scheduling-events` (排班与队列主题)
* **生产者：** 排班与队列微服务
* **消费者：** 门诊工作台微服务
* **核心事件：**
  * `QueueUpdatedEvent`（队列变更：如患者现场签到、线上取消挂号、医生呼号、患者过号等，门诊端消费后实时刷新医生工作台的患者候诊列表）。

## 7. 可观测性架构设计

### 7.1 双支柱全景

```
[各微服务]
  │  @Timed / @Observed
  │  SLF4J JSON 结构化日志
  │
  ├─── 链路追踪 ──➔ [OpenTelemetry Collector]
  │                      │
  │                      ▼
  │                  [Zipkin] ── 可视化调用拓扑 + 耗时瀑布图
  │
  └─── 日志 ────➔ [Fluentd] ──➔ [Elasticsearch] ──➔ [Kibana]
                    (容器日志采集,   (按 traceId      (统一日志检索)
                     JSON 解析,      关联检索)
                     traceId 注入)
```

### 7.2 链路追踪：跨服务 `traceId` 传播

微服务调用链中 `traceId` 的三条传播路径必须完整覆盖，否则追踪断链：

| 传播路径 | 实现方式 |
|---|---|
| **HTTP (Gateway → 微服务 → 微服务)** | Spring Cloud Gateway 生成 `traceId`，OpenFeign 拦截器自动注入 `X-B3-TraceId` 请求头 |
| **Kafka 消息** | 生产者将 `traceId` 写入 Kafka 消息头（`spring.kafka.producer.headers`），消费者自动提取并关联到当前 Span |
| **WebSocket (AI 助理)** | STOMP 连接握手时携带 `traceId`，AI 助理在 STOMP 消息头中透传，关联文本处理与推送链路 |

### 7.3 关键告警规则

| 告警项 | 条件 | 通知方式 |
|---|---|---|
| 医嘱提交失败 | `order_submit_error_rate > 1%` 持续 2min | 钉钉/企业微信 @值班医生 |
| Kafka 消费积压 | `kafka_consumer_lag > 1000` 持续 5min | 钉钉/企业微信 |
| 门诊服务不可用 | `/actuator/health` 连续 3 次失败 | 钉钉/企业微信 @运维 |
| AI 助理响应超时 | `ai_assistant_p95_latency > 3s` 持续 5min | 钉钉/企业微信 |

### 7.4 可观测性基础设施与业务服务解耦

- 所有可观测性组件（Zipkin、Elasticsearch、Kibana、Fluentd）与业务微服务在同一 `docker-compose.yml` 中定义，通过注解/配置自动集成。
- 业务代码仅需引入 `spring-boot-starter-actuator` 和 `micrometer-tracing-bridge-otel` 两个依赖，不感知收集器地址和存储后端，保持业务逻辑纯净。

## 8. 极简部署拓扑 (Docker Compose 方案)

由于移除了 K8s，整个医生端微服务集群及中间件可以通过单个或少数几个 `docker-compose.yml` 文件进行定义。

* **网络层：** 利用 Docker 内置的 `bridge` 网络（如 `cloud-brain-network`），各微服务容器通过容器名直接与 Nacos、MySQL 和 Kafka 互通，安全且无需向公网暴露中间件端口。
* **托管演进：** 若希望进一步降低机房运维成本，此结构可无缝平移至 **Zeabur** 等轻量级容器 PaaS 平台。代码提交后自动触发容器构建与滚动更新，平台自带的私有内网可直接承载 Nacos 与 Kafka 的流量，提供接近零运维的微服务体验。
