# 智慧云脑诊疗平台 - 需求分析图表指南

本文档梳理需求分析阶段需要绘制的 6 类图表，按从宏观到微观的顺序编排。

---

## 1. 业务模块图（顶层全景）

**目的：** 一张图说清平台的业务模块划分和用户角色归属。

**结构：三层布局**

- **顶层 — 用户角色：** Patient / Doctor / Technician / Admin
- **中层 — 业务模块：**
  - 患者端：AI 智能问诊 → 线上挂号
  - 医生端：门诊工作台 → 电子病历 → 医嘱/处方
  - 医技端：检查检验 → 影像上传 → AI 报告
  - 管理端：AI 排班 → 号源管理
- **底层 — 支撑系统：** AI 助理医生、药房管理、计费系统

**配色：** 蓝线（患者入流线）、绿线（医生门诊线）、红线（智能化医技线）、橙线（运筹调度线）。

---

## 2. 业务流程图（泳道图）

**目的：** 按角色分泳道，画出每条业务线的端到端流程。

**四条泳道，对应 `全流程描述.md` §1 的四条业务线：**

| 泳道 | 角色 | 流程节点 |
|---|---|---|
| 蓝线（诊前准备） | Patient | 注册 → 登录 → AI 智能分诊 → 线上挂号 → 进入候诊池 |
| 绿线（诊中枢纽） | Doctor | 登录 → 叫号接诊 → 问诊（AI 实时伴诊）→ 写病历 → 开检查/处方 → 结束就诊 |
| 红线（异步算力） | Technician + AI | 执行检查 → 上传影像（DICOM/NIfTI）→ AI 影像识别 → 生成结构化报告 → 回传门诊 |
| 橙线（后勤基座） | Admin | 登录 → AI 排班 → 释放/更新号源 → 出诊计划生效 |

**泳道交叉处的关键交互：**

- 蓝→绿 交汇点：线上挂号 + 线下挂号 → **队列合并** → 医生工作台统一叫号
- 绿→红 交汇点：医生下达检查医嘱 → **同步确认接收** → 异步触发检查
- 红→绿 交汇点：AI 报告生成 → 事件回传 → 医生工作台**红点/弹窗提示**
- 绿→蓝 交汇点：医嘱路由 → 药房配药 → 患者取药

## 3. 用例图

**目的：** 明确每个角色的功能边界和操作权限。

**4 个 Actor 及其用例：**

| Actor | 用例 | 优先级 |
|---|---|---|
| **Patient（患者）** | AI 智能问诊、查看检查报告、线上挂号、取消挂号、查看就诊记录 | Must-have |
| **Doctor（医生）** | 叫号接诊、读写电子病历、开具检查单、开具处方、结束就诊、查阅 AI 报告 | Must-have |
| **Technician（医技人员）** | 上传影像/检验数据、填写检查记录、查阅检查申请单 | Should-have |
| **Admin（管理员）** | AI 排班管理、号源管理、医生权限分配、系统配置 | Must-have |

**高风险用例（需额外标注）：** 开具处方、开具检查单——涉及患者安全，需要同步确认 + 幂等 + 二级密码验证。

## 4. 事件风暴

**目的：** DDD 需求分析核心工具，梳理领域事件、命令、聚合和限界上下文。

**步骤：**

1. **橙色事件贴纸** — 列出所有业务事件（对应 `技术栈与配置索引.md` §5 的 6 个 Kafka Topic）
2. **蓝色命令贴纸** — 什么操作触发了事件（如"医生点击开具医嘱"→ `OrderIssuedEvent`）
3. **黄色聚合贴纸** — 事件归属哪个聚合根
4. **紫色限界上下文** — 聚合归入哪个微服务

**核心事件链（一次完整就诊）：**

```
QueueUpdated ──▶ ConsultationStarted ──▶ OrderIssued ──▶ OrderFulfilled
                                                               │
                                          CtReportGenerated ◀──┘
                                                               │
                                          ConsultationAiSummary ◀── 结束就诊
                                                               │
                                          ConsultationFinished ◀────┘
```

**事件与命令对照表（部分）：**

| 命令 | 触发事件 | 所属聚合 | 所属上下文 |
|---|---|---|---|
| 医生叫号 | `ConsultationStartedEvent` | Consultation | 门诊工作台 |
| 开具医嘱 | `OrderIssuedEvent` | MedicalOrder | 医嘱路由 |
| AI 报告生成 | `CtReportGeneratedEvent` | ImagingTask | 医学影像 |
| 处方执行完成 | `OrderFulfilledEvent` | MedicalOrder | 医嘱路由 |
| 预约取消 | `QueueUpdatedEvent` | Queue | 排班队列 |
| 就诊结束 | `ConsultationFinishedEvent` | Consultation | 门诊工作台 |
| AI 摘要归档 | `ConsultationAiSummaryEvent` | DiagnosticSession | AI 助理 |

## 5. 状态机图

**目的：** 标注核心聚合根的状态流转规则，确保一致性。

### 5.1 就诊记录（Consultation）

```
WAITING ──(医生叫号)──▶ IN_PROGRESS ──(结束就诊)──▶ FINISHED
                           │
                           └──(患者过号)──▶ PASSED
```

- 仅 `WAITING` 状态可以进入 `IN_PROGRESS`
- 仅 `IN_PROGRESS` 状态可以进入 `FINISHED`
- 终态：`FINISHED`、`PASSED`（不可逆）

### 5.2 医嘱单（MedicalOrder）

```
PENDING ──(持久化确认)──▶ ACCEPTED ──(异步路由分发)──▶ ROUTED ──(下游完成)──▶ COMPLETED
                              │                            │
                              └──────────(驳回)────────────┘
                                                            ▼
                                                       REJECTED
```

- 同步阶段：`PENDING → ACCEPTED`（医生得到即时确认）
- 异步阶段：`ACCEPTED → ROUTED → COMPLETED`（事件驱动执行）
- 异常路径：任一阶段可因库存不足、禁忌症等原因跳转 `REJECTED`

### 5.3 影像分析任务（ImagingTask）

```
RECEIVED ──(预处理)──▶ PREPROCESSING ──(张量转换完成)──▶ INFERRING ──(模型推理完成)──▶ COMPLETED
                           │                                     │
                           └─────────────(失败)───────────────────┘
                                                                  ▼
                                                               FAILED
```

### 5.4 诊室队列（Queue）

```
WAITING ──(签到确认)──▶ IN_QUEUE ──(医生叫号)──▶ IN_CONSULTATION
                            │                         │
                            │                         ├──(正常结束)──▶ COMPLETED
                            │                         │
                            └──(患者取消)──▶ CANCELLED └──(患者过号)──▶ PASSED
```

## 6. 上下文映射图

**目的：** DDD 战略设计核心产出，画出限界上下文之间的集成关系。

**6 个限界上下文及其关系：**

```
                    ┌──────────────┐
                    │   认证授权    │
                    │  (通用域)     │
                    └──────┬───────┘
                           │ [共享内核 — JWT principal 注入请求头]
              ┌────────────┼────────────┐
              ▼            ▼            ▼
    ┌──────────────┐ ┌──────────┐ ┌──────────┐
    │  门诊工作台   │ │ 医学影像  │ │ 排班队列  │
    │   (核心域)    │ │ (支撑域)  │ │ (支撑域)  │
    └──┬───┬───┬──┘ └────┬─────┘ └────┬─────┘
       │   │   │         │             │
       │   │   │    [上游/下游]         │
       │   │   │  CtReportGenerated    │
       │   │   │         │             │
       │   │   │         ▼             │
       │   │   │  ┌──────────┐         │
       │   │   │  │  门诊工作台│◄────────┘
       │   │   │  └──────────┘    [上游/下游]
       │   │   │                   QueueUpdated
       │   │   │
       │   │   │         [同步 RPC]               [防腐层 ACL]
       │   │   └──────────▶ 医嘱路由 ──────────────────▶ 药房
       │   │                                   ────────▶ 检查科室
       │   │                                   ────────▶ 计费系统
       │   │
       │   │         [WebSocket 实时]           [Kafka 异步]
       │   └──────────▶ AI 助理医生 ◄──────────── AI 助理医生
       │                (文本+推送)            (摘要归档)
       │
       └────────────────────────────────────────────▶ 医嘱路由
                     ConsultationFinishedEvent
```

**关系标注说明：**

| 关系类型 | 上下文对 | 通信方式 |
|---|---|---|
| 共享内核 | 认证授权 → 所有上下文 | HTTP Header（`X-User-Id`, `X-User-Roles`） |
| 上游/下游 | 门诊工作台 → 医嘱路由 | **同步 RPC**（医嘱确认）+ Kafka 事件（后续执行） |
| 上游/下游 | 医嘱路由 → 药房/检查科室 | 防腐层 ACL + Kafka 事件 |
| 上游/下游 | 医学影像 → 门诊工作台 | Kafka 事件（`CtReportGeneratedEvent`） |
| 上游/下游 | 排班队列 → 门诊工作台 | Kafka 事件（`QueueUpdatedEvent`） |
| 双向实时 | 门诊工作台 ↔ AI 助理 | WebSocket（文本上行 + 建议推送下行） |
| 异步归档 | AI 助理 → 门诊工作台 | Kafka 事件（`ConsultationAiSummaryEvent`） |

```
① 业务模块图   →   ② 泳道流程图   →   ③ 用例图
        │                                │
        └────── ④ 事件风暴 ◄─────────────┘
```
